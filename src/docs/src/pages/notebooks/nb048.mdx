## Abstract

The canonical PING training recipe presents one MNIST digit per 200 ms trial. This entry asks whether the trained network — *without retraining* — can classify a *stream* of digits presented sequentially for $\tau$ ms each, far shorter than the trained $T$. The headline result: at $\tau = 50$ ms per digit (≈ 2 gamma cycles), the network classifies five sequential MNIST digits 5/5 correctly using only sparse cycle-locked spiking; the readout's class probability flips within one cycle of each digit transition. Sweeping $\tau$ down to 25 ms (one gamma cycle) with rate-compensated input gives 93% per-segment accuracy — **higher than the trained 200-ms baseline**, suggesting the trained readout's information per cycle is well-defined and the network is faster than its training duration suggests.

## Methods

**Network.** The canonical nb025 PING baseline at $\theta_u =$ off, seed 42, medium tier (1600 train / 400 test, 100 epochs). No retraining. Inference-only; the weights are read off disk and replayed.

**Sweep recipe (inference-only):**

| Parameter | Value |
|---|---|
| Integration timestep $\Delta t$ | 0.1 ms (matches trained config) |
| Trained $T$ | 200 ms |
| Stream segment duration $\tau$ | swept ∈ {25, 50, 100, 200} ms |
| Digits per stream | 5 (one segment per digit) |
| Streams per τ | 20 (small tier) / 80 (medium) — defaults to small |
| Input | uniform Poisson on 784 channels, 25 Hz per channel |
| Network | trained PING, medium-tier baseline (1600 samples, 100 epochs) |

**Stream construction.** For each $\tau$, sample $n$ MNIST test digits, encode each as a Poisson spike train for $\tau$ ms at the input rate, and concatenate along the time axis to form a single $(n\tau / \Delta t, 1, N_\text{in})$ input. Digit transitions are *hard switches* — the Poisson rate flips instantaneously at $t = k\tau$. The trained network processes the full stream in one forward pass; we override $M.T_\text{steps}$ to the stream length.

**Two input-rate protocols** isolate the temporal-integration effect from the drive-magnitude effect:

- **Constant input** — Poisson rate fixed at 25 Hz regardless of $\tau$. As $\tau$ shrinks, total spikes per digit drop proportionally. Accuracy degradation here conflates two effects.
- **Rate-compensated** — Poisson rate = $25 \cdot (200/\tau)$ Hz so total spikes per digit ≈ constant. Isolates the *temporal-integration* component of accuracy degradation.

**Readout.** The trained network uses a *mem-mean* readout: the output LIF's membrane potential is integrated over the trial and divided by $T_\text{steps}$ at the end. For streaming, we replay the same output LIF dynamics post-hoc on the recorded hidden spike train, then apply a **sliding-window mean** of width $\tau$ ms instead of the full-trial integration. At every timestep $t$:

$$
v_\text{out}(t) \;=\; \beta_\text{out}\, v_\text{out}(t-1) \;+\; \frac{1-\beta_\text{out}}{\Delta t}\, \mathbf{s}^E(t-1)\, W_\text{out},
\qquad
\text{logits}(t) \;=\; \frac{1}{w}\sum_{u=t-w+1}^{t} v_\text{out}(u)
$$

with $\beta_\text{out} = \exp(-\Delta t / \tau_\text{out})$, $\tau_\text{out} = 2$ ms (trained value), and $w$ the window length in steps. Per-segment accuracy is read off at the end of each $\tau$-window: $\text{pred}(k\tau - 1)$ vs the true label of segment $k$.

This is the *only* difference from the training-time readout — same $W_\text{out}$, same output LIF, same $\beta_\text{out}$. The sliding-window replaces full-trial mean averaging.

|---|---|---|---|

Two findings:

1. **The constant-input degradation isn't about timing — it's about drive magnitude.** At $\tau = 25$ ms with 25 Hz input, each digit gets only ≈ 0.5 input spikes per channel — much less drive than at training time. Rate-compensating ($\tau = 25$ ms × 200 Hz input) restores the total per-digit drive, and accuracy climbs to 93%.
2. **The trained 200-ms condition is not the optimum.** Rate-compensated $\tau = 25$ ms gives 93% on the headline-baseline checkpoint — 10 pp above the trained $\tau = 200$ ms point. Per-cycle information density at the rate-compensated cliff exceeds what a 7-cycle integration provides at the canonical input rate.

### Varying $(\tau, \text{rate})$ stream

The headline above held $(\tau, \text{rate})$ constant within a single stream and only varied them across streams. A stronger test: vary both knobs *within* a single stream — let each digit have its own segment duration and input rate, and check that the network still classifies each segment correctly.

| τ \ rate | 5 Hz | 10 Hz | 25 Hz | 50 Hz | 100 Hz | 200 Hz |
|---|---|---|---|---|---|---|
| 25 ms | 28% | 40% | 54% | 60% | 84% | 94% |

Three observations from the heatmap:

1. **The accuracy contour roughly follows the product $\tau \cdot \text{rate}$.** Both knobs control total per-digit input spikes (which scales as $\tau \cdot \text{rate} \cdot N_\text{in}$); the network responds to that integrated quantity. The roughly L-shaped 80%-iso-contour confirms this.
2. **The trained $(200 \text{ ms}, 25 \text{ Hz})$ cell sits at 90% — *interior* to the high-accuracy region**, not on its boundary. The architecture has substantial headroom in both directions.
3. **Off-axis "weak-drive" cells collapse below 50% only when both knobs are simultaneously squeezed.** Even $\tau = 25$ ms with 50 Hz, or $\tau = 100$ ms with 5 Hz, both clear 60%. The dynamics tolerate either short time or weak drive — but not both.

## Discussion

The headline figure makes four claims simultaneously visible: (a) sparseness — the E raster shows clear gamma-band structure, not asynchronous firing; (b) cycle locking — the I population fires in bursts at ≈ 28 ms cadence, identical to the trained dynamics; (c) functional encoding — the readout probabilities are not noise but track the true-class signal cleanly; (d) streaming inference — the readout *re-identifies* on each digit switch within ≈ 1 cycle, demonstrating that the cycle-locked spike pattern carries class information at the per-cycle timescale.

The $\tau$-sweep result has a more specific implication for the rate-floor mechanism story in [ar009](/articles/ar009/) / [ar010](/articles/ar010/). The architecture's per-cell rate is bounded by $r_E = p \cdot f_\gamma$ ([nb041](/notebooks/nb041/), [nb046](/notebooks/nb046/)), meaning each cell emits at most one spike per gamma cycle and the bulk-mean participation probability is $p \approx 0.2$. Streaming says: if the network can classify a digit in one cycle (rate-compensated 25 ms = 93%), the information per cycle is *sufficient* to support the classification task. The bound is tight but not crippling — the architecture isn't trading accuracy for sparsity, it's just performing the task at the temporal limit set by its own cycle clock.

Two caveats:

- **Sliding-window readout is not the trained readout.** It's the same output LIF and the same $W_\text{out}$, but the integration window is $\tau$ ms instead of $T$ ms. This is the minimal change needed to make a per-segment prediction; in principle one could retrain a streaming-specific readout to do better, but the point of this experiment is to show that the *already-trained* network is functional in the streaming regime without any new gradient.
- **Hard digit switches.** Real continuous-input regimes (speech, video) cross-fade rather than hard-switch. We don't test that. The hard switch is a stress test of the network's transient response.

## Next steps

- **Train on streaming data and compare.** Does a network trained with $\tau = 50$ ms segments and a sliding readout outperform the post-hoc streaming on the medium-tier baseline?
- **Cross-fade input transitions.** Replace the hard switch with a 10-ms linear blend between digits; does accuracy hold? Quantify the network's transient-response time directly.
- **τ × $\tau_\text{GABA}$ interaction.** The headline result uses the trained $\tau_\text{GABA} = 9$ ms. If $\tau_\text{GABA}$ is altered (per [nb041](/notebooks/nb041/)), the cycle period changes — does the streaming sweet-spot move with it? Affine-law prediction: optimal $\tau$ should scale linearly with the cycle period.
