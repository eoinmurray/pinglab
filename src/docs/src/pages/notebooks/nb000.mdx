## Introduction

Performance reference workload. The goal across the lab is **one architecture that runs efficiently on local Apple Silicon (MPS) and on Modal**, with a single code path — no device-specific branches, no CUDA-only fast paths, no MPS-only fallbacks. This notebook is the fixed point that work toward that goal is measured against: a recipe-identical clone of [nb006](/notebooks/nb006/) (standard-snn LIF classifier on MNIST at [*dt*](/the-oscilloscope/#network-flags-shared) = {cfg.dt ?? "?"} ms) trained through the shared [oscilloscope *train*](/the-oscilloscope/#subcommand-train) path. Holding the recipe constant means wall-time and throughput numbers reported here over successive runs reflect changes in the training stack itself — kernel launches, sync points, compile coverage, memory layout — not recipe drift.

Sibling per-model trainers ([nb006](/notebooks/nb006/)–[nb010](/notebooks/nb010/)) own the science. This entry owns the profile.

The three customisations match [nb006](/notebooks/nb006/)'s. Briefly:

- [**readout = li**](/the-oscilloscope/#readouts) — leaky-integrator readout; the default *rate* readout saturates CE on the forward pass at *T* = 2000 steps.
- [**surrogate-slope = 1**](/the-oscilloscope/#network-flags-shared) — bounds the fast-sigmoid pseudo-derivative; default slope = 5 overflows fp32 in *W_ff* gradients across 2000 BPTT steps.
- [**kaiming-init (signed)**](/the-oscilloscope/#network-flags-shared) — gives the hidden layer enough starting drive to spike at all.

### Performance

export const baselines = numbers.perf_baselines ?? (numbers.perf ? { current: numbers.perf } : {});
export const baselineRows = Object.entries(baselines);
export const fmtSps = (v) => (typeof v === "number" ? v.toFixed(1) : "—");
export const fmtSec = (v) => (typeof v === "number" ? v.toFixed(2) + " s" : "—");
export const fmtMem = (v) => {
  if (typeof v !== "number") return "—";
  if (v >= 1e9) return (v / 1e9).toFixed(2) + " GB";
  if (v >= 1e6) return (v / 1e6).toFixed(1) + " MB";
  return v + " B";
};

Each row is the most recent run on that backend; running again on the same backend updates that row, runs on a new backend append. Per-epoch breakdown (train compute, eval, observe) is in *metrics.json*; aggregates above are warm — they exclude the cold first epoch where allocator and one-shot JIT costs land.

Caveat on **Peak mem**: CUDA reports peak across the run via *max_memory_allocated* (counts autograd-saved intermediates). MPS reports a per-epoch sample of *current_allocated_memory*, which does not fully account for autograd-saved tensors — so the MPS number is reliably an *under*-estimate. Cross-backend comparison of peak memory is not meaningful.
