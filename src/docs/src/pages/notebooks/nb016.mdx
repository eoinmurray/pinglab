## Introduction

Single-MNIST training shows the network can recognise one digit. This entry asks what it costs to recognise a *stream* of them. The trial is always 200 ms; the trained single-digit readout is reused frozen, applied window by window. We sweep three axes — window length, input spikes per window, hidden spikes per window. The companion entry [nb015](/notebooks/nb015/) shares these weights and asks the dual question — *single*-window classification latency — including ping's autocorrelation analysis. Chance is 10%; the single-digit ceiling is $\approx$88%.

## Method

cuba, COBA, ping trained on standard single-MNIST per [nb010](/notebooks/nb010/), [nb011](/notebooks/nb011/), [nb012](/notebooks/nb012/), then frozen. Each trial is 200 ms; each window holds a freshly Poisson-encoded MNIST digit. Per-window predictions come from finite-differencing the trained mem-mean accumulator across window boundaries and argmax-ing through the same *W_out*. 800 trials per operating point. Three sweeps:

1. **Window length.** 200 ms / *n* with *n* ∈ {}, 25 Hz input.
2. **Input rate.** 4 × 50 ms layout, per-pixel rate ∈ {} Hz.
3. **Hidden spikes.** Re-projects sweep 1 onto measured hidden spike count per window.

## Success criteria

- ✅ All three recover their single-digit baseline at 200 ms (cuba {}%, COBA {}%, ping {}%) and degrade smoothly as the window shrinks; lowest tested point is ping at 5 ms ({}%, still above chance).
- ❌ ping loses its lead below 20 ms and breaks first at 5 ms — recurrent dynamics that help at long windows hurt at short ones.
- ✅ Per hidden spike, ping is $\approx$8× more efficient than cuba: {}% on $\sim${} spikes vs cuba {}% on $\sim${} (50 ms window).
- ⚠ Readout needs $\approx$25–50 input spikes per window to lock on; below that everything collapses toward chance, cuba most gracefully thanks to dense intrinsic activity.

## Next steps

- Train on the sequential task with a last-window loss; expect ping's short-window curve to flatten once the recurrent state is shaped by the task.
- Move to a task that forces window interaction (sum or compare two digits) so integration time becomes the bottleneck.
- Vary total trial length (100 ms, 400 ms) to separate "network refresh time" from "readout evidence time".
