## Introduction

Single-MNIST training tells you whether a network can recognise *one* digit; it says nothing about how the network's internal state evolves over the trial. This entry runs an *eval-only* probe on top of standard cuba / coba / ping training: every test trial is a sequence of four random MNIST digits, each shown for 50 ms inside a 200 ms trial. The trained mem-mean readout is queried as-is, and we ask — for each window position k — how often does the network's argmax prediction equal the digit that was on screen during window k?

A network whose state turns over fast (small effective τ relative to the 50 ms window) should peak at *k = 3*: the most recent digit dominates the readout. A network with slow recurrent persistence — most obviously PING with its E↔I loop — should be flatter or biased toward earlier windows.

## Method

Three networks (cuba, coba, ping) trained on standard single-MNIST at the configured tier; recipes inherited verbatim from [nb010](/notebooks/nb010/), [nb011](/notebooks/nb011/), [nb012](/notebooks/nb012/). After training, weights are frozen and a sequential-eval phase runs on the MNIST test set: 800 trials, each constructed as four randomly-drawn test images Poisson-encoded for 50 ms each, concatenated into a single 200 ms input. The model's mem-mean readout (taken over the *entire* 200 ms — same readout the network was trained with) yields one argmax prediction per trial. Per-target-window accuracy is the fraction of trials whose prediction equals the digit shown in window *k*, for *k* ∈ {0, 1, 2, 3}.

| param | value |
|---|---|
| windows × window length | 4 × 50 ms |
| total trial | 200 ms |
| trials | 800 |
| input rate | 25 Hz Poisson per pixel |
| readout | mem-mean over full 200 ms (unchanged from training) |

## Discussion

The flat-ish profile is the dominant feature: with a readout that averages over the entire trial, every window contributes roughly equally, so no single window dominates. The mild rightward tilt for ping ({(ping.window_3 - ping.window_0)?.toFixed?.(1) ?? "?"} percentage-point advantage of *w₃* over *w₀*) suggests its recurrent dynamics still let recent input dominate slightly — but the effect is buried under the trial-average readout.

The clean version of this experiment requires either:

1. **Tail-only readout**: average mem only over the last 50 ms window. Should sharpen *w₃* dramatically across all models.
2. **Per-window readout**: argmax at the end of each window separately, reporting accuracy at each position.
3. **Training on the sequential task**: with a tail-only loss, networks can *learn* to gate earlier windows out — the test of temporal capacity rather than passive dynamics.

Each adds complexity (custom readout flag in the oscilloscope, or a sequence-aware loss). The flat-readout numbers above are the cheapest probe; the next entry will pick one of these to actually separate the architectures.
