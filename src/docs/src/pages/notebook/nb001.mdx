# snnTorch calibration

<p><span class="font-mono">nb001</span> · Friday, 17<sup>th</sup> April 26</p>

| Run ID | Date | Size | Duration |
|---|---|---|---|
| <span class="font-mono">{nbRunId}</span> | {latestRun} | {cfg.tier ?? "?"} | {runDuration} |

The repo ships two variants of a snnTorch-style network: the in-repo *snntorch* model and *snntorch-library*, which swaps in snnTorch's own *snn.Leaky* + fast-sigmoid surrogate as a parity reference ([Eshraghian et al. 2023](https://arxiv.org/abs/2109.12894); [Neftci et al. 2019](https://arxiv.org/abs/1901.09948)). They are meant to be behaviourally equivalent at matched config — any residual gap localises to the LIF update and the surrogate-gradient details. Both models were trained on a {cfg.max_samples ?? "?"}-sample subset of MNIST for {cfg.epochs ?? "?"} epochs, batch size {cfg.batch_size ?? "?"}, lr {cfg.lr ?? "?"}, using signed Kaiming-uniform feedforward init (*--kaiming-init*), dt {cfg.dt ?? "?"} ms, T {cfg.t_ms ?? "?"} ms, {cfg.n_hidden ?? "?"} hidden units; only the *--model* flag differs between the two runs.

At matched config, the two implementations diverge sharply. Pinglab's *snntorch* reached {snn.best_acc ?? "?"}% test accuracy at epoch {snn.best_epoch ?? "?"}, while *snntorch-library* peaked at {lib.best_acc ?? "?"}% at epoch {lib.best_epoch ?? "?"} — a {ratio(snn.best_acc, lib.best_acc)}× accuracy ratio. Final-epoch training loss runs in the opposite direction: {num(snn.final_loss)} for the pinglab path against {num(lib.final_loss)} for the library path, a {ratio(lib.final_loss, snn.final_loss)}× gap with the library sitting higher. The two networks are not calibrated yet — a drop-in claim requires matching behaviour within a small tolerance, not the gap shown here.
