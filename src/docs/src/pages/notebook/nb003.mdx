*snntorch-clone* is pinglab's hand-rolled SNN; *snntorch-library* swaps in snnTorch's own *snn.Leaky* + fast-sigmoid surrogate as a parity reference ([Eshraghian et al. 2023](https://arxiv.org/abs/2109.12894); [Neftci et al. 2019](https://arxiv.org/abs/1901.09948)). They are meant to be behaviourally equivalent at matched config — any residual gap localises to the LIF update and surrogate gradient. Only the *--model* flag differs between runs; shared config is in the [Parameters](#parameters) table.

## Comparison of the models

*snntorch-library* is an external oracle — matching against it certifies that pinglab's LIF step and surrogate are equivalent to the canonical snnTorch package. The in-house *snntorch-clone* is not a duplicate of the oracle; it is rung zero of the [CUBA ladder](/models/#the-ladder) (*snntorch-clone → cuba → cuba-exp → coba → ping*), where each rung adds one biophysical feature. LIF code, surrogate, init, readout, and Poisson encoder stay fixed across rungs, so a gap between adjacent rungs localises to the single knob that changed.

Using the library directly as rung zero would conflate three changes between *library* and *cuba*: LIF implementation, surrogate, and the biophysical knob. nb003's narrow job is certifying the rung-zero calibration against the oracle so every downstream comparison stays single-variable.

## Role of *dt*

Both networks run at *dt* = {cfg.dt ?? "?"} ms for {cfg.t_ms ?? "?"} ms per sample, so $T_\text{steps} = T_\text{ms}/dt$. *dt* enters the LIF decay $\beta = e^{-dt/\tau_\text{mem}}$ with $\tau_\text{mem}$ = 10 ms — $\beta$ rescales with *dt* to keep the membrane time constant physical — and the Poisson encoder $p = r \cdot dt / 1000$, keeping expected spike count per ms invariant. What *dt* does not do is scale the drive: in canonical discretisation $V_{t+1} = \beta V_t + W s_t + b$ each kick is unit-size per step, so halving *dt* doubles kicks per ms. *cuba* corrects this with $(1-\beta)/dt$ drive scaling; *snntorch-clone* deliberately does not. The *patch_dt* helper in *src/pinglab/inputs.py* is the single refresh point for $T_\text{steps}$, $\beta$, and $p$.

The [snnTorch tutorial 5](https://snntorch.readthedocs.io/en/latest/tutorials/tutorial_5.html) does not carry *dt* at all — time is a step index, $\beta$ is a hardcoded 0.95, and $T_\text{steps}$ is chosen directly. That framing carries an implicit $dt = 1$ ms when mapped to physical time: $\beta = 0.95$ gives $\tau_\text{mem} \approx 19.5$ ms. At $dt = {cfg.dt ?? "?"}$ ms the same constant would imply $\tau_\text{mem} \approx 4.9$ ms, so pinglab recomputes $\beta$ from a physical $\tau_\text{mem}$ instead of reusing the tutorial's numeric constant.

## *firing_rates.png*

{hasFiringRates && <img src="/figures/notebook/nb003/firing_rates.png" alt="Hidden-layer firing rate vs epoch" />}

Mean hidden-layer firing rate (Hz) per epoch. A residual $\sim$2-3× gap remains in the steady-state rate — *snntorch-clone* settles consistently above *snntorch-library* across the tail of training, with within-model epoch-to-epoch std small enough (~0.1-0.3 Hz) that the gap is clearly systematic rather than noise. Both models converge to the same sparse CE attractor with accuracy matched to within ~2%, so the gap localises to what still differs after the $\tau$ match: the surrogate gradient shape (custom *SurrogateSpike* vs *fast_sigmoid*), the LIF-step ordering inside the autograd wrapping, and different RNG consumption orders during init (same distribution, different realisations, since the two classes allocate tensors in different sequences). Separating these would want a one-off diagnostic with weights copied across at init — a sidecar test rather than an ongoing notebook.
