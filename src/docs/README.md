## 🚀 Project Structure

Inside of your Astro project, you'll see the following folders and files:

```text
/
├── public/
├── src/
│   └── pages/
│       └── index.astro
└── package.json
```

Astro looks for `.astro` or `.md` files in the `src/pages/` directory. Each page is exposed as a route based on its file name.

There's nothing special about `src/components/`, but that's where we like to put any Astro/React/Vue/Svelte/Preact components.

Any static assets, like images, can be placed in the `public/` directory.

## 🧞 Commands

All commands are run from the root of the project, from a terminal:

| Command                   | Action                                           |
| :------------------------ | :----------------------------------------------- |
| `bun install`             | Installs dependencies                            |
| `bun dev`             | Starts local dev server at `localhost:4321`      |
| `bun build`           | Build your production site to `./dist/`          |
| `bun preview`         | Preview your build locally, before deploying     |
| `bun astro ...`       | Run CLI commands like `astro add`, `astro check` |
| `bun astro -- --help` | Get help using the Astro CLI                     |

## 🖼 Figure publishing convention

Published figures (for paper.md etc.) are *frozen copies* of the PNGs
generated by the harness under `src/artifacts/<experiment>/figures/`.

Flow:

1. Run the experiment: `uv run python src/pinglab/harness.py <experiment> experiment`
2. Inspect figures: `src/artifacts/<experiment>/figures/*.png`
3. When ready to publish, freeze the figure with provenance metadata via
   `src/scripts/freeze-figure.py`:

       uv run python src/scripts/freeze-figure.py <source> <dest>

   The script copies the PNG and writes a sidecar `<dest>.json` with the
   git SHA at freeze-time, the producing run's config (run_id, model, dt,
   samples, epochs), and the source artifact path. The destination depends
   on the figure's audience:
   - Paper-level figures (cited from `paper.md`):
     `src/docs/public/figures/<experiment>/`
   - Journal-entry figures (cited only from one journal entry):
     `src/docs/public/figures/<experiment>/journal/<entry-slug>/`
4. Reference in `src/docs/src/pages/<experiment>/...` via web path:
   - `![...](/figures/<experiment>/dt_sweep_combined.1000.40.png)`
   - `![...](/figures/<experiment>/journal/<entry-slug>/parity_rasters.png)`
5. Commit the markdown edits, the PNGs, and the sidecar JSON together —
   this freezes the paper's figures against the exact run they describe,
   and makes the provenance auditable from the repository state.

Namespacing journal-entry figures under their entry slug keeps the
top-level figures dir to actual paper exhibits, and lets a journal entry
be pruned (or its figures regenerated) without touching paper figures.
If a journal-entry finding is later promoted into the paper, re-run
`src/scripts/freeze-figure.py` to copy the figure up to the top-level dir with a
fresh sidecar (and a stable filename).

The harness is deliberately doc-agnostic — it writes to `src/artifacts/`
and never touches `src/docs/`. The freeze step is explicit, so figures
can't silently change in a published paper when the harness re-runs, and
the sidecar JSON makes "which run produced this figure?" a one-line
question rather than a manual archaeology session.
